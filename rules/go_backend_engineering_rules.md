# Go 项目规则

适用于以 Go 为核心、包含前后端协作、数据库、部署与运维要求的真实工程项目。

## 1. 总体原则

默认遵循：
- KISS
- YAGNI
- SOLID
- DRY

要求：
- 优先解决真实业务问题，不做炫技式设计
- 优先可维护、可测试、可观测，而不是短期堆代码
- 优先渐进式演进，避免大爆炸改造
- 优先复用现有能力，避免重复造轮子

## 2. 目录与分层规范

推荐分层：
- `api/` 或 `handler/`：协议适配、参数解析、响应封装
- `service/`：业务编排、流程控制、事务边界
- `repository/`：数据库或缓存访问
- `model/` 或 `domain/`：领域对象、数据结构、核心语义
- `provider/` 或 `adapter/`：外部服务接入层
- `middleware/`：通用中间件
- `config/`：配置加载与管理
- `pkg/`：可复用公共包
- `internal/`：仅项目内部使用的实现

约束：
- `handler` 不承载复杂业务逻辑
- `repository` 不承载业务编排
- `service` 不直接处理 HTTP 协议细节
- `utils` 只能放真正通用的无业务工具函数
- 禁止把复杂业务逻辑堆在 `main.go`、路由注册处或中间件中
- 依赖方向应尽量单向，避免循环依赖

## 3. 命名规范

- 包名短小、语义明确、小写
- 文件名体现职责，避免模糊命名
- 函数名表达动作和意图
- 变量名表达业务含义，避免 `data`、`info`、`tmp` 这类泛化命名
- 接口以能力命名，而不是以 `I` 前缀命名
- 错误变量和错误信息要能体现上下文

## 4. API 设计规范

### REST API
要求：
- 路径语义清晰，资源导向
- 请求与响应结构稳定
- 字段命名统一
- 明确分页、过滤、排序、鉴权和错误码
- 保持向后兼容，避免随意破坏旧接口

### 响应规范
- 成功响应与失败响应结构尽量统一
- 错误信息对前端友好，对日志可追踪
- 区分用户可见错误与系统内部错误
- 敏感错误不要直接暴露给客户端

### 幂等与重试
对以下场景优先考虑幂等：
- 支付
- 创建订单
- 状态变更
- 重复回调
- 用户重复点击提交

## 5. 错误处理规范

- 错误必须向上返回并保留上下文
- 不要吞错
- 不要只返回 `err` 而没有业务语义
- 对外错误信息与内部错误日志要分层
- 可以包装错误，但不要过度包装导致信息混乱

建议：
- 日志里记录根因、关键参数、请求链路标识
- 对外返回稳定错误码和可理解错误描述

## 6. 日志与可观测性规范

日志要求：
- 关键链路必须有日志
- 错误日志必须有上下文
- 使用统一日志格式
- 不打印敏感信息
- 避免无意义重复日志

建议记录：
- trace_id
- request_id
- user_id
- device_id
- session_id
- 核心业务标识

可观测性要求：
- 关键接口有耗时监控
- 外部依赖调用有成功率和延迟监控
- 错误率、超时率、重试率可观测
- 关键业务流程有埋点

## 7. 并发与 context 规范

- 请求链路必须尽量透传 `context.Context`
- goroutine 必须有明确生命周期
- 资源必须可释放
- channel 只在真正合适时使用，避免滥用
- 并发访问共享数据时必须考虑锁、原子操作或隔离策略
- 任何后台任务都要考虑超时、取消和异常恢复

禁止：
- 无控制地启动 goroutine
- 不可停止的后台任务
- 忽略 context cancellation
- 因并发导致数据竞态和资源泄漏

## 8. 数据库规范

### 表设计
- 表结构语义清晰
- 核心字段有明确约束
- 合理设置默认值、唯一约束和外键策略
- 明确软删除还是硬删除

### SQL 与查询
- 避免 N+1
- 避免无索引过滤
- 避免全表扫描型热点查询
- 分页优先考虑性能和稳定性
- 大事务要谨慎控制范围
- 批量操作优先使用批处理策略

### 索引规范
- 索引必须服务于真实查询路径
- 不要盲目堆索引
- 联合索引要结合 where / order by 场景设计
- 定期审视冗余索引和低效索引

### 迁移规范
- 数据库变更必须可审计
- 优先采用显式迁移脚本
- 迁移必须考虑回滚方案
- 大表变更必须评估锁表、耗时和风险
- 禁止在业务代码中偷偷执行 DDL

## 9. 配置管理规范

- 配置集中管理
- 区分开发、测试、生产环境
- 敏感配置通过安全方式注入
- 配置项要有默认值或校验机制
- 启动时尽早暴露配置错误

禁止：
- 硬编码密钥、地址、账号
- 在多个地方散落读取配置

## 10. 测试规范

### 单元测试
- 核心业务逻辑应可单元测试
- 表驱动优先
- 测试覆盖正常路径、边界路径和异常路径

### 集成测试
- 关键接口、数据库交互、外部依赖适配建议有集成测试
- 必要时使用测试容器或隔离环境

### 回归测试
以下改动优先补测试：
- 核心业务逻辑
- 并发逻辑
- 数据库查询逻辑
- 权限与鉴权逻辑
- 修复过线上事故的代码

## 11. 前后端协作规范

- 接口字段语义先定清楚再开发
- 明确可空字段、默认值和枚举值
- 前端必须处理加载态、错误态、空态和权限态
- 后端必须保证错误码和错误响应稳定
- 涉及状态变更的接口优先考虑防重和幂等

## 12. 安全规范

必须默认考虑：
- 鉴权
- 权限校验
- 输入校验
- SQL 注入
- XSS / CSRF 风险
- 敏感信息脱敏
- 文件上传安全
- 第三方回调验签
- 速率限制

禁止：
- 信任前端传入的关键状态
- 在日志中输出敏感数据
- 把内部错误栈直接暴露给客户端

## 13. 发布与回滚规范

上线前至少确认：
- 配置正确
- 数据库迁移已评估
- 核心接口已验证
- 监控与告警已准备
- 回滚方案明确
- 外部依赖可用性已确认

高风险改动建议：
- 灰度发布
- 开关控制
- 分阶段放量
- 保留回滚路径

## 14. 重构规范

- 重构先定义目标，不要为了“看起来更优雅”而重构
- 优先局部渐进式重构
- 新旧逻辑切换要可验证、可回滚
- 大型重构优先先抽边界，再迁移实现
- 重构中不要顺手掺杂无关改动

## 15. 排障规范

排障时优先顺序：
1. 先看日志和监控
2. 先确定是否为变更引起
3. 先缩小范围，再深入定位
4. 优先验证最可能根因
5. 修复后补监控、日志和测试，防止复发

不要：
- 没有证据就乱改代码
- 用临时 patch 掩盖根因
- 忽略复盘和防复发措施
