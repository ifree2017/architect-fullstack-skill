# Web 前端专项规则

适用于 Web 管理后台、ToB 平台、ToC H5、官网、活动页、SSR/CSR 前端项目。目标是保证 Web 前端在页面交互、状态管理、性能、权限、联调和上线方面更加专项化，而不是停留在泛前端规则层面。

## 1. 总体原则

默认遵循：
- 优先完整用户流程，而不是单页面局部正确
- 优先可维护、可测试、可观测
- 优先与设计系统、现有路由和权限体系一致
- 优先渐进式演进，避免页面和状态失控

## 2. Web 场景重点

Web 前端相比通用前端，要额外关注：
- 浏览器兼容性
- 路由与权限
- 页面首屏与资源加载
- SEO（如适用）
- CSR / SSR / SSG 差异
- 大列表、大表单、大后台页面性能
- 多标签页、多窗口、刷新恢复能力
- H5 在微信/企业微信/内嵌 WebView 中的行为差异

## 3. 页面与路由规范

- 页面应围绕完整业务流程设计
- 路由层明确公开页、登录页、受保护页、无权限页
- 面包屑、菜单、页面标题、权限点要语义一致
- 刷新页面后状态恢复策略要明确
- 避免路由切换时出现权限闪烁或重复请求

## 4. SSR / CSR / SEO 规则

### 如果是 CSR 后台系统
重点关注：
- 权限加载
- 首屏骨架
- 大资源拆包
- 首次 API 并发策略

### 如果是 SSR / SEO 场景
重点关注：
- 页面可索引内容
- Meta 信息
- 服务端渲染与客户端 hydration 一致性
- 避免仅靠客户端请求后才出现核心内容

## 5. Web 性能专项

优先关注：
- 首屏加载时间
- JS 包体积
- 路由级拆包
- 静态资源缓存
- 列表渲染性能
- 大表单重渲染
- 富文本、图表、复杂组件懒加载
- 重复请求和无效轮询

建议：
- 路由级代码分割
- 大组件懒加载
- 表格和列表按需渲染
- 对高频输入、搜索、筛选做防抖/节流
- 合理缓存 API 请求结果

## 6. 浏览器与容器兼容性

必须考虑：
- 主流浏览器支持范围
- 移动端 H5 与桌面浏览器差异
- 微信内 H5 与系统浏览器差异
- 文件上传、下载、复制、粘贴在不同环境的行为差异
- 日期、时区、滚动、键盘弹起等兼容性问题

## 7. 后台系统专项规则

对于后台/中台类页面，重点关注：
- 列表页查询、筛选、分页、导出
- 权限控制与按钮级权限
- 大表单回填和校验
- 弹窗与抽屉逻辑不要失控
- 批量操作要有确认和幂等意识
- 导入导出要考虑异步化、任务化和错误反馈

## 8. H5 专项规则

对于 ToC/H5 页面，重点关注：
- 首屏速度
- 分享能力
- 登录态恢复
- 页面来源场景识别
- 弱网与中断恢复
- 埋点与转化漏斗
- 在微信、企业微信、App WebView 下的行为差异

## 9. 联调与接口协作

- 前端字段语义必须先和后端对齐
- 错误码要可稳定处理
- 权限失效、登录过期、重复提交要统一处理
- 对导出、上传、轮询、异步任务状态查询要有统一封装

## 10. 上线专项检查

上线前至少确认：
- 路由权限正常
- 关键页面首屏正常
- 错误态、空态、无权限态验证完成
- 生产环境 API 域名和配置正确
- 埋点、监控、错误上报已接入
- 关键浏览器和典型容器已验证

## 11. 默认评审问题

每次评审 Web 前端实现时，至少问：
- 这是否覆盖了完整用户流程
- 路由和权限是否可靠
- 首屏和大页面性能是否可接受
- 在目标浏览器/容器下是否可靠
- 联调错误处理是否统一
- 上线后是否容易监控和排障
