# API 设计规则

适用于 REST API、WebSocket API、内部服务接口和可对外开放的 OpenAPI 风格接口。目标是保证接口语义稳定、易于协作、便于演进、可观测、可维护，并降低前后端联调和线上变更风险。

## 1. 总体原则

默认遵循：
- 语义优先于实现细节
- 稳定性优先于局部方便
- 兼容性优先于随意修改
- 一致性优先于个人习惯
- 可演进优先于一次性拼凑

要求：
- API 要表达清晰的业务语义，而不是暴露内部实现
- 接口设计先定契约，再推进前后端开发
- 同类接口保持路径、字段、错误码、分页和鉴权风格一致
- 不为了当前页面省事而破坏长期演进能力

## 2. 接口分类规范

建议明确区分：
- 对外开放 API
- 前后端交互 API
- 内部服务 API
- WebSocket 实时 API
- 异步回调 / webhook API

要求：
- 不同类型接口的鉴权、错误码和 SLA 预期要明确
- 对外 API 与内部 API 不要混用同一套隐式约定
- 回调类接口要特别强调幂等、验签和重试兼容

## 3. REST 路径设计规范

### 路径要求
- 路径表达资源语义
- 使用名词而不是动词为主
- 层级不宜过深
- 命名统一、稳定、可读

建议：
- `/users`
- `/users/{id}`
- `/orders/{id}/items`

避免：
- `/getUserInfo`
- `/doCreateOrder`
- `/queryAllThingsByCondition`

### 方法语义
- `GET`：查询
- `POST`：创建 / 复杂查询 / 非幂等动作
- `PUT`：整体更新
- `PATCH`：局部更新
- `DELETE`：删除

如果动作为主而不是资源 CRUD，也应保证语义明确，例如：
- `POST /orders/{id}/cancel`
- `POST /devices/{id}/bind`

## 4. 请求与响应结构规范

### 请求结构
必须明确：
- 必填字段
- 可选字段
- 默认值
- 可空字段
- 枚举值范围
- 时间格式
- 金额、数量、分页参数的单位和精度

### 响应结构
要求：
- 成功响应结构尽量统一
- 错误响应结构尽量统一
- 字段命名稳定
- 不暴露内部实现字段
- 对外字段不要频繁改名

建议统一包含：
- 业务数据
- 请求追踪标识（如适合）
- 分页信息（列表场景）
- 稳定错误码（失败场景）

## 5. 字段设计规范

- 字段名必须表达业务语义
- 布尔字段要清楚表达真假含义
- ID 字段命名保持一致
- 时间字段统一使用明确格式
- 枚举字段要定义完整枚举值及含义
- 不要使用模糊字段如 `status2`、`typeFlag`、`data`、`info`

注意：
- 字段一旦对外暴露，变更成本很高
- 字段设计先考虑兼容性，再考虑短期方便

## 6. 分页、过滤、排序规范

### 分页
必须明确：
- 页码分页还是游标分页
- 默认页大小
- 最大页大小
- 返回总数是否保证准确

建议：
- 小中型列表可用 page + page_size
- 大数据量场景优先考虑 cursor / seek 分页

### 过滤与排序
- 过滤字段语义明确
- 排序字段应受控，不要任意拼接
- 复杂检索建议封装清楚，不把查询协议设计成随意透传 SQL 风格

## 7. 错误码规范

### 基本要求
- 错误码稳定
- 错误码分层
- 错误描述对调用方可理解
- 内部异常细节不要直接暴露

建议区分：
- 参数错误
- 鉴权失败
- 权限不足
- 资源不存在
- 状态冲突
- 幂等冲突
- 频率限制
- 外部依赖失败
- 系统内部错误

### 错误设计要求
- 相同类型错误尽量返回相同语义错误码
- 前端可根据错误码做稳定处理
- 日志中要能映射到更详细的内部错误上下文

## 8. 鉴权与权限规范

必须明确：
- 接口是否需要登录
- 是否需要设备、租户、用户维度鉴权
- 是否需要角色 / 权限点校验
- 鉴权失败与权限不足如何区分

不要：
- 把权限校验完全交给前端
- 不同接口对同一权限规则表现不一致

## 9. 幂等与重复请求规范

这些场景必须优先考虑幂等：
- 创建订单
- 资金变更
- 状态迁移
- 回调通知
- 用户重复点击提交
- 网络重试场景

要求：
- 明确幂等键
- 明确幂等窗口
- 明确重复请求的返回语义
- 不要让重复请求产生脏数据或重复副作用

## 10. 兼容性与版本化规范

### 兼容性原则
- 尽量新增字段，不轻易删除字段
- 尽量保证旧调用方继续可用
- 修改字段语义前必须评估所有调用方
- 返回值结构不要随意破坏

### 版本化
在这些场景下考虑版本化：
- 字段语义发生根本变化
- 无法兼容旧行为
- 对外开放 API 且调用方不可控

版本化要求：
- 版本策略一致
- 旧版本生命周期明确
- 升级路径明确

## 11. WebSocket 设计规范

适用于实时交互场景。

必须明确：
- 连接建立流程
- 鉴权方式
- 心跳机制
- 消息类型
- 请求响应关联方式
- 服务端主动推送格式
- 中断与重连策略
- 错误消息格式

要求：
- 消息类型要稳定
- 字段语义要清楚
- 区分客户端请求、服务端事件、系统通知、错误消息
- 避免协议无版本、无类型、无状态定义

## 12. 回调 / Webhook 规范

- 必须支持验签或等价安全机制
- 必须考虑重复回调
- 必须有幂等处理
- 明确重试语义
- 返回结果要便于对方判断是否需要重试
- 记录完整调用日志便于追查

## 13. 文档与协作规范

接口至少应清楚描述：
- 路径和方法
- 请求参数
- 响应结构
- 错误码
- 鉴权要求
- 示例
- 幂等说明
- 限流规则（如有）
- 兼容性注意事项

前后端联调前：
- 字段先对齐
- 枚举先对齐
- 错误码先对齐
- 时间和金额格式先对齐

## 14. 可观测性规范

接口应具备：
- request_id / trace_id
- 关键路径日志
- 错误日志上下文
- 耗时监控
- 成功率、超时率、错误率监控
- 外部依赖调用监控

## 15. 默认评审问题

每次评审 API 设计时，至少问：
- 语义是否清晰
- 是否一致
- 是否兼容
- 是否容易联调
- 是否支持演进
- 是否考虑鉴权、幂等、错误码和监控
- 是否会让前后端产生理解歧义
