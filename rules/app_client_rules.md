# App 客户端专项规则

适用于 iOS / Android 原生 App、Hybrid App、内嵌 H5 容器、以及需要和移动端客户端深度协作的场景。目标是补齐 App 端特有的版本兼容、发布审核、推送、埋点、崩溃、Deep Link、设备标识和容器协作问题。

## 1. 总体原则

- 把 App 视为长期在线、版本碎片化、发布受审核和分发约束的客户端
- 优先考虑版本兼容性，而不是假设所有用户都在最新版
- 优先考虑登录态、设备标识、推送、埋点和崩溃恢复
- Hybrid / H5 场景要明确容器能力边界

## 2. App 特有关注点

必须额外关注：
- iOS / Android 差异
- 版本兼容
- 灰度发布与渠道差异
- 应用审核与上架节奏
- 推送
- Deep Link / Universal Link / Scheme
- 登录态续期
- 设备标识
- 崩溃、埋点和性能监控
- 前后台切换和进程重启恢复

## 3. 版本兼容规则

- 后端接口不要默认只服务最新版本 App
- 字段新增优先向后兼容
- 行为变化要考虑老版本客户端
- 不兼容改动要有版本门槛、降级或强制升级策略
- 与客户端对齐最小兼容版本

## 4. 登录态与设备标识规则

重点关注：
- Token 过期与续期
- 多设备登录策略
- 设备标识使用边界
- 登录失效后的回退路径
- 冷启动恢复登录态
- 账号切换时本地缓存清理

原则：
- 登录和设备相关逻辑不能靠前端“猜”
- 服务端应提供稳定、可恢复的鉴权机制

## 5. Deep Link / Scheme / Universal Link 规则

- 页面路由和外部唤起要有统一语义
- 链接参数要稳定可追踪
- 未登录用户承接流程要明确
- App 未安装时的降级策略要明确
- 不同平台跳转行为要分别验证

## 6. 推送与消息规则

- 区分通知点击、静默推送、站内消息
- 推送到达 ≠ 用户看到 ≠ 用户点击
- 推送落地页必须可恢复上下文
- 推送相关状态变化要与服务端对齐
- 重要消息要考虑幂等和重复送达问题

## 7. Hybrid / WebView 协作规则

如果 App 内嵌 H5 或 Hybrid：
- 明确 JSBridge 能力边界
- 明确哪些能力由容器提供，哪些由 H5 实现
- 登录态同步机制要明确
- 页面返回、关闭、分享、支付、上传等交互要统一设计
- WebView 缓存、版本差异和调试能力要提前考虑

## 8. 崩溃、埋点与性能观测

必须尽量具备：
- 崩溃上报
- ANR / 卡顿监控（Android）
- 启动耗时
- 页面打开耗时
- 登录和支付转化漏斗
- 网络失败率
- 推送到达与点击埋点
- 关键功能成功率

## 9. 发布与审核规则

- iOS / Android 发布节奏要分开考虑
- 应用商店审核时间和审核风险要提前评估
- 渠道包差异、灰度比例、热更新策略要明确
- 高风险功能建议加开关控制
- 与服务端联动改动要考虑客户端升级滞后

## 10. API 协作规则

App 场景接口设计要额外关注：
- 弱网和离线重试
- 幂等提交
- 大版本兼容
- 分页与缓存策略
- 上传下载稳定性
- 登录态过期处理
- 异步任务状态恢复

## 11. 安全规则

重点考虑：
- 登录凭证保护
- 设备指纹边界
- 越狱 / Root 风险感知（如有要求）
- 本地敏感信息存储
- H5 容器与原生通信安全
- Deep Link 参数校验
- 支付和交易安全

## 12. 默认评审问题

每次评审 App / 移动端协作方案时，至少问：
- 是否考虑旧版本兼容
- 登录态和设备标识是否稳
- 推送 / 跳转 / 支付链路是否完整
- 崩溃、埋点、性能监控是否具备
- 发布审核与灰度是否考虑到
- Hybrid / WebView 边界是否清晰
