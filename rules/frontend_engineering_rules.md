# 前端工程规则

适用于 React / Vue / Next.js / 通用 SPA 或全栈前端项目，目标是保证页面交互、组件边界、接口协作、状态管理和上线质量都足够稳定、清晰、可维护。

## 1. 总体原则

默认遵循：
- KISS
- YAGNI
- SOLID
- DRY

要求：
- 优先解决完整用户流程，而不是只修页面局部
- 优先可维护、可读、可测试，而不是堆技巧
- 优先渐进式演进，避免组件和状态失控
- 优先与现有设计系统、组件规范和交互模式保持一致

## 2. 页面与组件设计规范

### 页面职责
页面层负责：
- 路由级数据组织
- 页面级状态编排
- 关键交互流程控制
- 调用业务 hooks / store / service

不要：
- 在页面里堆大量可复用的业务细节
- 让页面承担过多 UI 细节和重复逻辑

### 组件职责
组件按职责拆分：
- 展示组件：尽量纯展示，弱依赖业务
- 容器组件：负责状态和数据组织
- 业务组件：聚焦某个明确场景
- 通用组件：必须真正常用且抽象稳定后再沉淀

约束：
- 单个组件职责尽量单一
- 不要为了“复用”过早抽象出复杂组件
- props 命名要表达语义，不要含糊
- 组件接口要稳定，避免把内部实现细节暴露出去

## 3. 状态管理规范

状态分层建议：
- 本地 UI 状态：组件内部处理
- 页面级状态：页面或页面级 hooks 处理
- 跨页面共享状态：store 管理
- 服务端数据状态：优先交给专门的数据获取层处理

要求：
- 区分 UI 状态、业务状态、服务端状态
- 不要把所有状态都塞进全局 store
- 不要在多个地方维护同一份源数据
- 派生状态尽量从源数据计算，避免重复存储
- 状态更新路径必须清晰可追踪

## 4. 数据获取与 API 协作规范

### 请求规范
- 请求层统一封装
- 统一处理鉴权、超时、错误码、重试策略
- 明确区分业务错误与系统错误
- 对幂等提交、重复点击、并发请求冲突有防护

### 接口协作
前后端字段必须先对齐：
- 字段语义
- 可空性
- 默认值
- 枚举范围
- 错误码
- 分页、过滤、排序
- 鉴权行为

不要：
- 前端自行猜测接口字段含义
- 接口返回不稳定却在页面写死逻辑
- 多个页面各自实现不同的错误处理标准

## 5. 页面状态规范

任何关键页面至少考虑这些状态：
- 初始态
- 加载态
- 空态
- 成功态
- 错误态
- 权限不足态
- 部分加载失败态

表单页额外考虑：
- 默认值加载
- 提交中
- 提交成功
- 提交失败
- 重复提交
- 离开未保存提示

列表页额外考虑：
- 首屏加载
- 分页加载
- 筛选切换
- 列表为空
- 局部刷新
- 操作后数据同步

## 6. 表单规范

- 表单项语义清晰
- 校验规则前后端尽量一致
- 必填、格式、范围、依赖校验明确
- 错误提示要贴近字段且用户可理解
- 提交按钮在提交中应禁用或具备防重能力
- 有草稿、回填、重置、离开提示时要明确规则

不要：
- 把业务校验完全依赖前端
- 让用户在错误发生后看不懂怎么修正
- 出现重复提交导致脏数据

## 7. 路由与权限规范

- 路由层清楚区分公开页、登录页、受保护页
- 权限控制应有统一入口
- 页面级权限不足要有明确反馈
- 鉴权失效要有统一处理机制
- 重定向逻辑要避免循环跳转

## 8. UI 一致性规范

- 尽量复用设计系统或统一组件
- 相同交互模式保持一致
- 按钮、弹窗、表格、表单、提示样式统一
- 错误提示、成功提示、确认操作文案统一
- 响应式行为要符合页面预期

## 9. 性能规范

优先关注：
- 首屏加载体积
- 列表渲染性能
- 不必要的重复请求
- 不必要的重复渲染
- 大组件频繁更新
- 图片和资源加载优化

建议：
- 合理拆分组件
- 合理使用 memo / computed / cache
- 大列表考虑虚拟滚动
- 请求去重、缓存和预取按实际需要使用
- 懒加载用于真正有价值的场景

不要：
- 为了“优化”引入难以维护的复杂技巧
- 在没有瓶颈证据时过度优化

## 10. 错误处理规范

- 页面级错误要可感知
- 请求失败要有明确反馈
- 对局部失败和全局失败区别处理
- 系统错误文案不能把底层异常原样暴露给用户
- 日志和监控要能追踪错误上下文

## 11. 可测试性规范

优先保证这些可测试：
- 核心用户流程
- 表单校验逻辑
- 权限控制逻辑
- 关键组件渲染分支
- 异步请求成功与失败分支

测试分层建议：
- 单元测试：组件逻辑、hooks、工具函数
- 集成测试：页面交互流程
- E2E：核心业务主路径

## 12. 安全规范

必须默认考虑：
- XSS 风险
- CSRF 场景
- Token / Cookie 管理
- 敏感信息展示控制
- 文件上传校验
- 权限绕过风险
- 不信任前端本地状态作为唯一安全依据

## 13. 前端上线检查清单

上线前至少确认：
- 主流程已验证
- 错误态和空态已验证
- 权限控制已验证
- 提交防重已验证
- API 兼容性已确认
- 埋点、日志、监控已准备
- 回滚方式明确

## 14. 重构规范

- 先识别页面/组件膨胀的根因
- 优先拆边界，再拆实现
- 状态混乱时先厘清单一数据源
- 渐进式替换，不大爆炸重写
- 保持旧页面和新页面切换可控

## 15. 默认评审问题

每次评审前端实现时，至少问：
- 这是不是在解决完整用户问题
- 页面状态是否考虑完整
- 组件边界是否合理
- 状态是否单一可信
- 接口契约是否稳定
- 是否容易维护和测试
- 是否有性能或权限风险
