# 小程序专项规则

适用于微信小程序、支付宝小程序、抖音小程序等小程序场景。目标是补齐小程序特有的生命周期、登录态、审核、性能、支付、分享、订阅消息和发布规则。

## 1. 总体原则

- 把小程序看成独立客户端，不要简单当成“缩小版 H5”
- 优先考虑平台限制、生命周期和审核规则
- 优先保证首屏速度、登录稳定性和弱网体验
- 优先考虑分享、支付、订阅消息、场景值等平台特性

## 2. 小程序特有关注点

必须额外关注：
- App / Page 生命周期
- 登录态与 session 管理
- 场景值与来源识别
- 分享链路
- 支付链路
- 订阅消息
- 审核与发布流程
- 分包与包体积限制
- 宿主环境 API 限制

## 3. 生命周期规则

- 区分 App 生命周期与页面生命周期
- 页面返回前台、切后台、重新进入时要有恢复策略
- 登录态、页面状态、未完成任务在生命周期切换时要可恢复或可重试
- 避免页面卸载与返回栈逻辑混乱

## 4. 登录与鉴权规则

重点关注：
- 登录态过期
- session 续期
- 静默登录与显式登录边界
- 用户授权失败处理
- 多端登录和切账号逻辑
- openid / unionid / device_id 等标识的使用边界

要求：
- 登录链路要可重试
- 登录失败要有明确兜底和用户提示
- 不要把登录态逻辑散落在多个页面里

## 5. 场景值与分享规则

- 明确页面从哪里进入
- 区分扫码进入、分享进入、会话进入、历史记录进入
- 分享参数设计要稳定、可追踪
- 分享落地页要能正确恢复上下文
- 被分享页面要考虑未登录用户的承接流程

## 6. 支付与交易链路规则

涉及支付时必须考虑：
- 下单幂等
- 支付发起失败
- 支付结果回调延迟
- 用户取消支付
- 前端支付结果与后端真实结果不一致
- 补单与状态刷新

原则：
- 以前端结果为提示，以后端状态为准
- 关键订单状态必须可恢复、可重查

## 7. 订阅消息规则

- 区分一次性订阅与长期订阅（按平台规则）
- 用户拒绝后的降级策略要明确
- 订阅触发时机要合理，避免打扰
- 与后端消息发送条件要先对齐

## 8. 性能与包体积规则

小程序尤其要关注：
- 首包体积
- 分包策略
- 图片与资源大小
- 首屏接口并发数
- 大列表滚动性能
- 页面切换卡顿
- 低端机表现

建议：
- 核心路径优先轻量化
- 不把低频大能力塞进主包
- 复杂图表、富文本、重组件谨慎接入

## 9. 审核与发布规则

- 提前了解平台审核要求
- 敏感能力、支付、内容、用户信息收集要合规
- 审核环境和生产环境差异要明确
- 发布前确认小程序配置、域名、隐私协议、备案信息等

## 10. API 协作规则

- 接口设计要考虑弱网和重试
- 登录相关接口要特别稳定
- 状态变更接口优先考虑幂等
- 文件上传、支付、分享落地、消息状态查询等要专项测试
- 错误提示要更贴近用户，不要只暴露系统报错

## 11. 埋点与观测规则

建议重点记录：
- 启动来源
- 首屏耗时
- 登录成功率
- 支付发起率 / 成功率
- 分享点击和回流
- 关键转化漏斗
- 接口错误率
- 页面异常

## 12. 默认评审问题

每次评审小程序实现时，至少问：
- 生命周期切换是否可靠
- 登录态是否稳
- 分享 / 支付 / 订阅链路是否完整
- 包体积和首屏性能是否可接受
- 审核和发布限制是否考虑到
- 弱网、返回前台、重新进入是否会出问题
