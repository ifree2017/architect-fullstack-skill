# 数据库工程规则

适用于 PostgreSQL 为主的业务系统，也可迁移到其他关系型数据库。目标是保证表结构、SQL、索引、迁移、性能、稳定性和上线风险都可控。

## 1. 总体原则

默认遵循：
- 优先正确性，再追求性能
- 优先可维护，再追求局部技巧
- 优先显式约束，避免隐式约定
- 优先可回滚、可审计、可观测

要求：
- 数据库设计服务于真实业务与查询路径
- 结构设计要考虑未来 3~12 个月演进
- 不在代码里偷偷做 DDL
- 任何高风险变更都必须有回滚方案

## 2. 表设计规范

### 基本要求
- 表名、字段名语义明确
- 主键策略统一
- 时间字段统一规范
- 核心字段尽量有 NOT NULL、默认值或合理约束
- 唯一性规则尽量通过唯一约束表达
- 明确软删除还是硬删除

### 字段设计
- 避免语义模糊字段
- 避免一个字段承载多个含义
- 枚举值要可维护，必要时单独建字典表或在代码中统一定义
- JSON 字段只在确实适合半结构化数据时使用
- 核心查询字段不要随意放进 JSON 里

## 3. 关系设计规范

- 明确一对一、一对多、多对多关系
- 多对多关系优先中间表建模
- 外键是否启用要结合项目策略，但逻辑约束必须明确
- 不要为了图省事把强关联关系设计得过于松散

## 4. 索引规范

### 索引设计原则
- 索引必须服务真实查询路径
- 联合索引顺序结合 where / order by / join 场景设计
- 不为低频查询堆过多索引
- 更新频繁表要控制索引数量

### 索引评估要点
新增索引前先确认：
- 解决的是哪个查询问题
- 命中率是否足够高
- 是否影响写入性能
- 是否已有可复用索引
- 是否会产生冗余索引

### 常见问题
警惕：
- 单列索引很多但查询总是多条件组合
- `like '%xxx'` 导致索引失效
- 隐式类型转换导致索引失效
- 排序和过滤字段顺序不合理
- 低选择性字段滥建索引

## 5. SQL 编写规范

### 查询
- 明确查询目标字段，避免无意义 `select *`
- 复杂查询先保证正确，再优化
- 尽量让 where 条件、join 条件、排序逻辑清晰
- 避免把业务逻辑塞进过度复杂 SQL 中导致难维护

### 更新
- update / delete 必须有明确条件
- 批量更新要控制范围和事务时间
- 状态变更优先带上前置条件，避免覆盖更新
- 涉及并发竞争时考虑乐观锁 / 悲观锁 / 版本号策略

### 分页
- 小数据量下可用 offset + limit
- 大偏移量分页要评估性能问题
- 大表深分页优先考虑基于游标或基于唯一键的翻页方案

## 6. 事务规范

- 明确事务边界，不要把无关操作塞进同一事务
- 事务尽量短
- 外部调用不要放在长事务中
- 评估锁竞争、死锁和回滚成本
- 高并发关键路径要特别关注隔离级别和锁策略

## 7. 数据一致性规范

默认思考这些问题：
- 是否允许最终一致性
- 是否需要幂等
- 是否需要补偿机制
- 是否可能出现重复写入
- 是否可能出现部分成功部分失败
- 状态机是否会出现非法跃迁

关键状态流转必须：
- 有明确前置条件
- 有并发控制策略
- 有异常恢复方案

## 8. 迁移规范

### 基本要求
- 所有结构变更通过显式迁移脚本管理
- 迁移脚本必须可审计
- 上线前评估执行耗时、锁表风险和数据影响范围
- 高风险迁移需要预演

### 大表变更原则
- 尽量拆成多步迁移
- 避免一次性高风险变更
- 新增列优先兼容旧逻辑
- 先双写/兼容，再切流量，再清理旧字段
- 大索引创建、字段类型变更、回填操作必须重点评估

### 回滚原则
- 每次迁移前明确回滚路径
- 回滚不仅是撤销脚本，还包括数据状态恢复策略
- 不可逆变更必须提前说明风险

## 9. 性能优化规范

### 优化顺序
优先顺序：
1. 查清瓶颈 SQL
2. 看执行计划
3. 看索引是否合理
4. 看是否需要改查询方式
5. 再考虑分库分表、缓存或归档

### 不要一上来就做的事
- 不要一有慢查询就立刻分库分表
- 不要没有证据就加缓存
- 不要没分析执行计划就盲调 SQL
- 不要把结构问题全甩给中间件或硬件

## 10. 缓存协同规范

如果数据库与缓存一起使用，必须明确：
- 缓存粒度
- 一致性要求
- 失效策略
- 热点 key 风险
- 缓存击穿、穿透、雪崩风险
- 更新后如何保证读到合理数据

不要：
- 没有一致性策略就随意引入缓存
- 把缓存当成掩盖坏 SQL 的唯一手段

## 11. 审计与观测规范

应尽量具备：
- 慢查询监控
- 锁等待监控
- 错误率监控
- 核心表读写量监控
- 索引使用情况分析
- 关键数据变更审计能力

高风险业务建议记录：
- 操作人
- 操作时间
- 操作对象
- 关键字段变更前后值

## 12. 安全规范

- 最小权限原则
- 不在应用中使用高危数据库账号
- 敏感数据加密或脱敏
- 导出能力要审计
- 避免在日志中输出敏感字段
- SQL 参数化，防止注入

## 13. 上线检查清单

每次涉及数据库变更上线前，至少确认：
- 迁移脚本已审核
- 执行顺序明确
- 大表风险已评估
- 索引影响已评估
- 回滚方案明确
- 依赖该变更的代码版本关系明确
- 监控和告警已准备

## 14. 默认评审问题

每次评审数据库设计时，至少问：
- 结构是否准确表达业务语义
- 查询路径是否清楚
- 索引是否真的解决问题
- 是否容易产生脏数据或重复数据
- 是否存在事务过大或锁竞争风险
- 是否考虑了迁移、回滚和未来演进
